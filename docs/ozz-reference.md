`Waifu-Animation` does not copy Ozz C++ implementation code. It intentionally follows these Ozz runtime concepts:

- `Skeleton` stores joint parent indices, joint names, and rest poses as separate arrays. Ozz stores rest poses in SoA form for performance; this library stores one transform per joint because JavaScript engines and VRM adapters consume AoS shapes more naturally.
- Skeleton utilities mirror Ozz's runtime helper surface in TypeScript form: `getJointLocalRestPose` returns a cloned local rest transform, `isLeaf` scans the parent-index hierarchy, and depth-first/reverse-depth-first generators yield joint indices, parent indices, and joint records across multiple roots or a selected subtree.
- `RawSkeleton` mirrors Ozz's offline/runtime split for skeleton authoring: callers can edit `roots` and each joint's `children` array directly, use `iterateRawSkeletonDepthFirst` / `iterateRawSkeletonBreadthFirst` for deterministic hierarchy walks, validate raw names/humanoid aliases/cycles before building, and call `SkeletonBuilder.build` or `buildSkeletonFromRawSkeleton` to produce a runtime `Skeleton`.
- The raw skeleton builder flattens roots and children in Ozz-style depth-first order, resolves each child parent by name through the existing `createSkeleton` path, clones rest transforms, and preserves parent-before-child runtime indices across multi-root hierarchies.
- `RawAnimation` mirrors Ozz's offline/runtime split for animation authoring: callers can edit per-joint raw tracks with `translations`, `rotations`, and `scales`, validate strict key timing/value rules, clone raw data without aliasing, and call `AnimationBuilder.build` or `buildAnimationFromRawAnimation` to produce a runtime `AnimationClip`.
- The raw animation builder validates skeleton mapping when a `Skeleton` is supplied, rejects duplicate resolved target channels, sorts emitted runtime tracks deterministically by skeleton joint or target name and TRS property, normalizes raw rotation keys, fixes adjacent quaternion signs for shortest-path interpolation, and never mutates the editable raw animation input.
- Raw animation utilities mirror Ozz's offline helper surface in TypeScript form: `extractRawAnimationTimePoints` returns deterministic unique sorted key times across raw translation, rotation, and scale channels with optional joint/property filtering; `createFixedRateSamplingTimes` produces bounded fixed-rate times and ratios from `0` to the sanitized duration; and `sampleRawAnimation` / `sampleRawAnimationAtRatio` evaluate raw TRS keys without mutating raw input. When a `Skeleton` is supplied, raw sampling validates target mapping and merges keyed raw properties onto a rest-pose output; without a skeleton it returns raw track-order transforms with identity defaults for unkeyed TRS properties.
- `AnimationOptimizer` mirrors the reusable core of Ozz's offline optimizer without claiming a C++ port: `optimizeRawAnimation` / `tryOptimizeRawAnimation` validate raw animation and skeleton mappings, return structured issues for invalid inputs, clone outputs, preserve first and last keys, remove interpolable translation/scale/rotation keys within local tolerances, and compare rotations with shortest-path quaternion angle error. `AnimationOptimizerOptions` supports base tolerances, per-joint tolerance/weight overrides by joint name, humanoid name, or joint index, and descendant-count hierarchy weighting for stricter parent-joint reduction. `compareAnimationSampleError` samples raw or runtime `AnimationClip` sources against a `Skeleton` and reports RMS/max translation, rotation, and scale errors for optimizer validation or visualization. This does not implement Ozz `.ozz` archive compatibility, importer config, a CLI, or the exact propagated model-space optimizer metric.
- `AdditiveAnimationBuilder` mirrors Ozz's offline additive builder as a TypeScript `AnimationClip` preprocessing helper: callers can build from each channel's first keyed sample or from an explicit reference pose, invalid clips/reference poses fail through structured issues, and output remains a normal rest-encoded `AnimationClip` that works with Waifu's existing runtime additive layer path.
- `Animation` separates clip metadata from translation, rotation, and scale key data. Ozz sorts runtime keyframes for cache-friendly sampling. This library stores browser-delivered keyframe payloads as versioned binary data, decodes them into typed arrays, validates them aggressively, and samples into local pose buffers.
- `PackedRuntimeAnimation` is the TypeScript-friendly optional analogue for Ozz's preprocessed runtime animation object: callers can build or try-build it from an `AnimationClip` (including clips produced by `AnimationBuilder`), validate it, inspect archive/version and key-controller stats, and sample it into a `Pose`. It clones source clip buffers, freezes the packed object graph, sorts key controllers deterministically by resolved joint/target and TRS property, stores global iframe times/ratios, and stores per-track lower/upper seek tables against those iframes.
- Sampling writes local-space transforms. It does not directly mutate renderer bones. `sampleClipToPose` preserves the existing seconds-based loop/clamp behavior, while `sampleClipToPoseAtRatio` mirrors Ozz `SamplingJob` ratio sampling by clamping ratios to `[0,1]`. `AnimationSamplingContext` is the TypeScript analogue of `SamplingJob::Context`: it keeps per-track lower-key intervals for coherent forward sampling, seeks when time moves backward or the clip changes, resizes to the clip's track count, and exposes a snapshot of cache reuse/search counters for diagnostics.
- `getAnimationClipStats` exposes runtime clip metadata similar to Ozz `Animation` inspection helpers: duration, channel track count, resolved joint count, SoA track count, unique timepoints, translation/rotation/scale key counts, and per-track key-controller details.
- Root motion follows the Ozz motion extraction/playback split: callers explicitly choose a carrier joint, can extract translation and yaw/full rotation into reusable generic motion tracks with `extractRootMotion`, and can optionally bake selected carrier motion out to a cloned in-place `AnimationClip`. `extractRawRootMotion` and `MotionExtractor.extractRaw` provide the same core behavior for editable `RawAnimation` data, returning a cloned raw in-place animation plus motion tracks. Loop distribution follows Ozz's per-channel approach by spreading the first/last translation or rotation difference over each extracted channel's keys instead of snapping only the endpoint. Runtime consumers can sample `MotionTracks`, compute signed interval deltas across looping or clamped ranges, and use `MotionAccumulator`/`MotionSampler` to accumulate motion outside renderer bone mutation. `AnimationRuntime.update(deltaSeconds, { collectRootMotion: true })` still exposes a blended override-layer motion delta without applying that motion to skeleton poses.
- Generic user tracks follow Ozz's raw-track/build/sampling split with TypeScript-friendly helpers. `getRawUserTrackStats` reports raw key and value-component counts, while `optimizeRawUserTrack` is an Ozz `TrackOptimizer`-inspired reduction pass that validates raw tracks, preserves names and step keys, removes identity/constant/interpolable linear keys within tolerance, and keeps shortest-path normalized quaternion behavior. This is reusable TypeScript code and is not copied from Ozz C++ source.
- Blending operates on local poses with explicit weights, masks, bind/rest fallback, quaternion shortest-path interpolation, and normalization. Per-joint mask weights follow Ozz semantics: negative and non-finite values are treated as zero, while positive values above `1` are preserved. `computeLocomotionBlendWeights` and `synchronizeLocomotionPlayback` expose the reusable core of the Ozz blend sample's walk/jog/run policy: triangular weights over an ordered clip set and playback speeds derived from weighted authored durations, with deterministic zero-weight and zero-duration handling.
- Additive layers are separate from ordinary override blending and are applied as deltas.
- Local-to-model conversion happens after local pose composition and before world-space constraints. `localToModelPose` updates the full hierarchy, while `updateLocalToModelPoseRange` mirrors Ozz `LocalToModelJob` root/from/to/from-excluded usage for deterministic child-range refreshes after procedural local pose edits.
- IK and aim/look-at are procedural jobs with explicit input/output boundaries, not scattered bone writes.
- Two-bone IK exposes correction quaternions for the first two chain joints, matching Ozz's separation between solving the chain and letting the caller apply corrections to the active pose.
- `solveTwoBoneIk` and `solveTwoBoneIkCorrections` preserve the original model-space position/correction API. `solveTwoBoneIkModel` is the Ozz-style matrix job wrapper: it accepts root/mid/end model matrices or decomposed model transforms, reports model-space deltas, and also returns local-space `rootLocalCorrection`/`midLocalCorrection` values to post-multiply into the active local pose.
- Aim IK accepts Ozz-style local forward/up/offset axes plus model-space joint and target inputs, but returns `jointCorrection` as a model-space delta for Waifu's correction adapters. Apply it before the current model rotation (`jointCorrection * jointModelRotation`); `targetDirection`, `offsettedForward`, `correctedForward`, `correctedUp`, and `alignmentError` are reported in that same model space.
- `applyAimIkModelCorrection` converts that model-space aim delta into a local post-multiply correction before writing to a pose, so `solveAimIk().jointCorrection` keeps its existing model-space semantics. `applyAimIkChainToPose` runs and applies aim jobs in the caller-provided joint order, refreshing model matrices after each correction. `applyAimIkChildToParentChainToPose` mirrors the Ozz look-at sample policy more closely by propagating the corrected child forward/offset into each parent joint's local space and refreshing model matrices once from the parent-most edited joint. `createHumanoidLookAtAimChain` provides a renderer-agnostic humanoid metadata preset for head/neck/chest/spine aim chains.
- Two-bone IK keeps soft/limited extension separate from physical reach clamping. A softened solve can leave the endpoint slightly short near full extension without reporting the target as unreachable.
- Foot IK follows the Ozz `samples/foot_ik` ordering at the planning layer: derive ankle targets from ground point/normal, compute pelvis compensation from the lowest foot correction, then solve per-leg IK with explicit clamping and skipped-contact reporting. `solveOzzFootIk` wraps that core with humanoid leg defaults, caller-provided raycast/contact floor sampling, default ray/foot-height semantics, and ankle aim corrections that align a configurable ankle-up axis to the floor normal. The TypeScript jobs return plans and correction quaternions for consumers instead of directly mutating renderer bones.

Intentional differences:

- The core is TypeScript and renderer-agnostic, so it avoids Ozz's C++ memory layout and SIMD-specific SoA implementation. Range-limited local-to-model updates operate on the library's AoS `Transform[]`/`Mat4[]` arrays rather than Ozz SoA spans.
- `AnimationBuilder` still outputs the existing `AnimationClip` runtime shape as the broad public interchange type. `PackedRuntimeAnimation` is available as an explicit preprocessing step when callers want immutable sorted key-controller metadata and iframe/seek tables. It intentionally remains a TypeScript/browser representation and does not mirror Ozz allocation, compression, SoA hot data, GV4 encoding, or `.ozz` binary archives.
- Ozz `IKAimJob` outputs a local-space correction to multiply with the joint local-space quaternion; `solveAimIk` converts that solve into a model-space correction so it matches Waifu's existing world-correction application path.
- Ozz `MotionExtractor` is an offline tool that can be wired from importer config and `.ozz` archives. Waifu implements the reusable in-memory TypeScript core for `AnimationClip` and `RawAnimation` inputs, but importer config, direct FBX/glTF property extraction, a CLI bridge, and `.ozz` archive compatibility remain outside browser core.
- Ozz raw skeleton and raw animation archives are not mirrored. Waifu's raw skeleton and raw animation APIs are in-memory TypeScript authoring surfaces and builder boundaries; `.ozz` archive import/export remains outside browser core.
- Ozz raw animation utility functions return booleans for invalid inputs and write into caller-owned spans. Waifu throws structured validation errors through the existing TypeScript validation path and allocates ordinary transform arrays for raw sampling results.
- JSON is metadata-only. Waifu ships animation keyframes as `.waifuanim.bin` payloads, decoded by `Waifu-Animation` into typed arrays before runtime sampling or Three.js binding.
- VRM humanoid naming is first-class in adapters and manifests, but generic skeleton math is not VRM-only.
- Visual capture and browser/WebGL integration remain in the Waifu application, not the reusable core.
